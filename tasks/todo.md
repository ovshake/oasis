# Crypto Exchange Simulation — Implementation Plan

Branch: `crypto-exchange` (based on `b94c510`, pre-stock-market, vanilla OASIS)

A narrative-driven cross-asset market simulation as a tool for traders to explore scenarios.

### Design invariants (already decided)
- **Fixed-tick simulation, 1 minute per step.** Every step, every agent is polled. The gate decides whether they act or remain silent. The step is the smallest time unit; all timestamps in the DB are step-indexed.
- **Synchronous agent evaluation within a step.** All agents decide based on the end-of-step-(N-1) snapshot — no agent in step N sees another agent's step-N action. This makes runs reproducible and parallelizable.
- **6 levers per persona**: system prompt, numeric params, action-tier gate, info filter, memory, hard invariants.
- **Persona card format**: Pydantic dataclass + YAML archetype templates + generated JSONL library.
- **Reflection**: out of MVP.
- **Gate-before-LLM**: silent agents skip the LLM call entirely (cost-savings layer).
- **Multi-asset wallet** via `balance(user_id, instrument_id, amount, locked)` — single-cash wallet is not enough.
- **Single generic `instrument` + `pair` schema** — no `company` / `stock_order` / `token` split.

### Time & step semantics

| Concept | Value |
|---|---|
| Tick length | 1 minute (sim time) |
| Smallest time unit | 1 step |
| Typical MVP scenario | 60–480 steps (1h–8h sim time) |
| Full-day scenario | 1440 steps |
| All DB `created_at` | `epoch + step * 60s`, fully derivable from `step` |
| Agent polling | 100% every step (gate may return `silent`) |

**The tick (order of operations within step N):**

1. Clock advances to step N.
2. Fire scheduled god-mode news events → `news_event` + `post` rows.
3. Snapshot state — all agents in step N see end-of-step-(N-1) state.
4. Per-agent: compute stimulus (price moves, news, followed-account activity, mentions, replies).
5. Per-agent: gate samples tier (`silent | react | comment | post | trade`) — pure Python, no LLM.
6. Active (non-silent) agents: build prompts in parallel.
7. Batched LLM calls → proposed actions.
8. Invariant check: reject actions violating hard rules.
9. Apply actions: posts/comments/likes/orders persisted with `created_at = step*60s`.
10. Matching engine: per-pair match-settle cycle inside a single transaction (self-trade guarded, escrow-based).
11. Auto-market-news if `|Δprice| > threshold` on any pair.
12. Update `pair.last_price`.
13. Write step's memory rows for all agents (active agents: their action; silent agents: passive feed observations).
14. Telemetry snapshot: prices, volumes, tier distribution, per-archetype P&L delta.
15. Clock advances to step N+1.

### Runtime cost envelope at 1-min resolution

| Scenario length | Steps | Active LLM calls (@5% active) | Cost on GPT-4o-mini | Wall-clock (@50 concurrent) |
|---|---|---|---|---|
| 1 hour | 60 | ~30k | ~$1–3 | ~5 min |
| 4 hours | 240 | ~120k | ~$6–15 | ~20 min |
| 8 hours | 480 | ~240k | ~$12–30 | ~40 min |
| 1 day | 1440 | ~720k | ~$30–75 | ~2 h |

Active rate spikes during news events (15–30%). Budget 2–3× these figures for eventful scenarios. MVP default: 240-step (4-hour) scenarios. Traders can scale up when something interesting emerges.

**Gate-only mode**: harness supports `--no-llm` flag that uses persona-driven heuristics (no LLM calls). Used for calibration and large-scale distribution validation. Essentially free.

## 3. Phases

Each phase is independently shippable with a verification step. Skip to Phase 10 at any point to validate end-to-end on a small toy.

### Phase 1 — Data model & package scaffold
- [ ] Create `oasis/crypto/` subpackage with empty modules + `__init__.py`.
- [ ] Write SQL schemas for: `instrument`, `pair`, `balance`, `order`, `trade`, `news_event`, `persona`, `agent_persona`, `agent_memory`.
- [ ] Add all proper indices on day 1 (`order(pair_id, side, status, price)`, `trade(pair_id, created_at)`, `agent_memory(user_id, step)`).
- [ ] Use `FOREIGN KEY` + `CHECK` constraints throughout; enable `PRAGMA foreign_keys = ON`.
- [ ] Extend OASIS's DB init to load these schemas when `crypto_enabled=True`.
- [ ] `data/market/assets.yaml` with static defaults for the 6 assets (symbol, name, class, yfinance_ticker, peg_target, decimals, default_price).
- [ ] Seed the 6 assets from `assets.yaml` at DB init.
- **Verify**: unit test creates a DB, seeds assets, reads them back, confirms constraints reject bad inserts.

### Phase 2 — Matching engine (with day-1 fixes)
- [ ] `Exchange.place_order(user_id, pair_id, side, price, qty, step)` — limit orders first; market orders phase 2.
- [ ] `Exchange.cancel_order(order_id)` — refund escrow to `balance.locked` → `balance.amount`.
- [ ] `Exchange.match_all_pairs(step)` — called ONCE per tick from the harness; iterates over all pairs.
- [ ] `Exchange._match_pair(pair_id, step)` — price-time priority, maker-price execution, partial fills.
- [ ] **Transaction wrapping**: every match-settle cycle runs inside a single SQLite transaction. Failure → rollback, not half-state.
- [ ] **Self-trade prevention**: skip-and-cancel policy; matching function checks `buyer_id != seller_id`.
- [ ] **Proper escrow**: buy → lock `price*qty` of quote asset; sell → lock `qty` of base asset.
- [ ] Multi-asset settlement: on fill, buyer's base `balance.amount` ↑, seller's quote `balance.amount` ↑, both `locked` ↓.
- [ ] Order timestamps and trade timestamps derive from `step` (no real-time).
- **Verify**: tests for matching, partial fill, self-trade guard, atomicity (simulated crash mid-match → tree stays consistent), escrow conservation (sum of all amounts + locked is invariant). Deterministic: same orders placed in same step produce identical trades regardless of submission order within the step.

### Phase 3 — Persona dataclass & archetype templates
- [ ] `Persona` Pydantic model with fields: id, archetype, name, backstory, voice_style, 5 numeric axes, action_base_rates, hard_rules, initial_holdings, follows_archetypes.
- [ ] `HardRule` model — name, condition (DSL string), effect (DSL string). Simple eval at decision time.
- [ ] 10 archetype YAML templates in `data/personas/archetypes/` with param ranges + base-rate tables from the plan.
- [ ] `PersonaLibrary` class — loads `library.jsonl`, samples by population-mix, resolves archetype template defaults.
- [ ] Tests: round-trip serialization, archetype validation, sampling respects mix.
- **Verify**: load a small fixture library, sample 100 personas with a given mix, confirm counts match within Poisson noise.

### Phase 4 — Persona generation pipeline (Sonnet 4.6)
- [ ] `scripts/generate_personas.py` — async batched generator using `claude-sonnet-4-6`.
- [ ] Prompt template: archetype card + diversity seeds (age bracket, region, profession, crypto entry year, primary platform).
- [ ] Batch 50 per call, async concurrency ~20, with retry+backoff on rate limits.
- [ ] Validation layer: reject personas with invalid params, duplicate names, missing required fields; re-request on failure.
- [ ] Output: `data/personas/library.jsonl` (1 persona per line for streaming).
- [ ] Generation metadata sidecar: `library.meta.json` (model, timestamp, prompt hash, counts per archetype, cost).
- [ ] **Budget estimate**: 10k personas × ~500 output tokens ≈ 5M output tokens on Sonnet 4.6. Plan: ~$100 one-time.
- [ ] Smoke run: generate 100 personas end-to-end first; inspect diversity by hand; then scale to 10k.
- **Verify**: library passes all validation; archetype distribution matches target within 1%; diversity check on names (no >2 duplicates) and on backstories (cosine-similarity histogram is broad).

### Phase 5 — Action gate (Lever 3)
- [ ] `ActionGate(persona)` class.
- [ ] **Invoked every step for every agent** — this is the mechanism that makes the fixed-tick design affordable.
- [ ] Base rates read from `persona.action_base_rates`.
- [ ] Stimulus formula: `w_price * |Δport| + w_news * news_indicator + w_follow * followed_activity + w_mention * mentions + w_personal * replies_to_me` — MVP keeps `w_price` and `w_news` non-zero, others = 0 for day 1.
- [ ] Adjusted `p_silent` via `p_silent ** (1 + α * stimulus * social_sensitivity)`, renormalize remaining tiers.
- [ ] `decide_tier(stimulus)` → `silent | react | comment | post | trade` — pure Python, no LLM, <1ms per call.
- [ ] Tier → allowed `ActionType` subset passed into the LLM prompt (for non-silent agents).
- [ ] Per-agent rate limiters (sliding window) as safety net.
- [ ] Vectorized variant: `ActionGate.decide_tiers_batch(personas, stimuli)` uses numpy so 10k-agent gating is <100ms per step.
- **Verify**: run 1000 agents × 100 steps in quiet market with gate only (no LLM); action histogram matches configured base rates within 2σ. Then run with LLM and confirm post volume dropped ~10× vs. no-gate baseline.

### Phase 6 — Information layer (Lever 4)

**Social graph topology: hybrid (persona-affinity × preferential attachment).**

- [ ] **Stage 1 — Persona-affinity filter.** For each new agent `a`, compute the eligible follow-pool: other agents whose archetype is weighted ≥ some threshold in `a.follows_archetypes`. A FOMO Degen's pool = mostly KOLs + News Traders + other FOMO Degens; a Lurker's pool = KOLs + a broad sample; an MM's pool = minimal (they don't care about social signal).
- [ ] **Stage 2 — Preferential attachment within the pool.** Sample `k` follow targets from the eligible pool with weights `w_i = follower_count_i + 1` (Barabási-Albert style). This produces power-law degree distributions with α ≈ 2.1 — top 2% (KOLs) accrue 95%+ of followers, as in real Twitter/CT graphs.
- [ ] **Target-count per archetype**: `k` drawn from persona-appropriate distributions:
  - Lurker: k ~ Uniform(1, 20)
  - HODLer: k ~ Uniform(10, 50)
  - FOMO Degen: k ~ Uniform(50, 200)
  - Paperhands: k ~ Uniform(20, 100)
  - TA: k ~ Uniform(10, 50)
  - Contrarian: k ~ Uniform(10, 40)
  - News Trader: k ~ Uniform(50, 300)
  - Whale: k ~ Uniform(5, 30)
  - KOL: k ~ Uniform(50, 200) (KOLs follow *other* KOLs — influencer-of-influencers dynamics)
  - MM: k ~ Uniform(5, 20) (minimal social usage)
- [ ] Build order matters for preferential attachment. Randomize agent ordering per-seed but log the seed for reproducibility.
- [ ] Feed filter: each persona has attention weights by (topic, sentiment, source archetype) → filters the step's post stream before the LLM prompt is built.
- [ ] TA nerds see price-action posts preferentially; KOL-followers see KOL posts; Whales see other whales' trade posts.
- [ ] Follow edges persisted to the existing OASIS `follow` table.
- **Verify**: (a) follow-graph degree distribution passes power-law KS test (p > 0.05); (b) top 2% of agents by follower count are >80% KOL or Whale archetype; (c) a FOMO Degen's follow-list contains ≥60% KOL/News-Trader archetypes; (d) two different-archetype agents at the same step have Jaccard overlap < 0.5 on their filtered feeds.

### Phase 7 — Short-term memory (Lever 5)
- [ ] `MemoryStore` class backed by `agent_memory` table.
- [ ] Writes: action taken, realized P&L delta, notable observations (posts from followed KOLs, own mentions).
- [ ] Reads: last 5 actions + current P&L summary + last K notable observations for the agent.
- [ ] Inject into prompt as `RECENT MEMORY:` block.
- **Verify**: integration test — agent makes a trade in step 3; at step 4 the LLM prompt includes that trade in the memory block.

### Phase 8 — Simulation harness (fixed-tick orchestrator)

This is the core of the simulation engine — a `Simulation` class in `oasis/crypto/harness.py` that owns the clock, agent pool, exchange, and news scheduler, and drives the per-step tick. It is the reusable library underneath the scenario CLI.

- [ ] `TickClock` in `oasis/crypto/clock.py`: fixed-interval clock with `current_step`, `step_to_datetime(step)`, `advance()`, configurable `start_datetime` and `step_minutes` (default 1).
- [ ] `Simulation(config)` class in `oasis/crypto/harness.py` with:
  - `run()` — main loop: for step in range(duration): tick(step).
  - `tick(step)` — executes the 15-step tick spec from §1 (Time & step semantics) in a deterministic order.
  - `_compute_stimuli()` — numpy-vectorized over all agents.
  - `_build_prompts(active)` — parallel per-agent prompt construction.
  - `_batched_llm(prompts)` — async concurrent LLM calls (configurable concurrency, default 50).
  - `_apply_action(agent, action, step)` — routes to post / comment / like / order.
  - `_update_memories(step)` — writes memory rows for every agent (active + silent).
  - `_snapshot(step)` — telemetry per step.
- [ ] `--no-llm` / `gate_only=True` mode: active agents take heuristic actions based on persona + invariants instead of calling the LLM. Used for calibration and distribution validation at full scale.
- [ ] `Telemetry` class in `oasis/crypto/telemetry.py`: buffers per-step rows in memory, flushes to parquet on completion. Schemas for `trades`, `orders`, `posts`, `prices`, `actions`, `stimuli`, `tiers`.
- [ ] Conservation invariants checked every N steps (configurable; default every 60): total cash + total inventory (valued at last_price) is constant modulo deterministic error.
- [ ] Reproducibility: single `seed` argument on `Simulation`, threaded through numpy/random and any stochastic persona decisions. LLM calls are still non-deterministic unless the model is deterministic.
- **Verify**: 1000-agent × 100-step simulation in `gate_only` mode completes in <30s wall-clock, produces the expected action-tier distribution per archetype, passes the conservation check, and produces identical output on two runs with the same seed.

### Phase 9 — Scenario YAML + CLI runner (incl. news ingestion)

News ingestion lives in `oasis/crypto/news_ingest.py` using an adapter pattern; the scenario CLI consumes it. Three news modes: `manual` (YAML inline), `historical` (fetch past date range), `live_snapshot` (fetch current).

**News ingestion module:**
- [ ] `NewsItem` dataclass — raw: timestamp, title, body, source, url. Enriched adds: sentiment, affected_assets, audience, magnitude, credibility.
- [ ] `NewsAdapter` base class with `fetch(date_range_or_now) -> list[RawNewsItem]`.
- [ ] `ManualAdapter` — reads `news_timeline` list from scenario YAML.
- [ ] `CryptoPanicAdapter` — GET `https://cryptopanic.com/api/v1/posts/` with `currencies`, `filter`, `public=true`. Free tier 30 req/min.
- [ ] `enrich_news(items) -> list[NewsEvent]` — Sonnet 4.6 batched call; returns structured NewsEvent per item.
- [ ] Cache layer: `data/market/news_cache/{date_range}.jsonl` — air-gapped reproducibility.
- [ ] Step-mapping: `wall_clock_to_step(timestamp, scenario_start, step_minutes)` — maps a news UTC timestamp to the sim step it fires in.
- [ ] `scripts/ingest_news.py` CLI — offline preprocessing of a date range (fetch + enrich + cache). Safe to run standalone so a trader can prepare news fixtures before sim.

**Scenario YAML schema (extended):**
- [ ] `name`, `duration_steps`, `step_minutes=1`, `seed`, `assets`, `price_source`, `as_of_date`, `initial_prices`, `population_mix`, `output_dir`, and:
- [ ] `news_source`: `{kind: manual | historical | live_snapshot, providers, date_range, lookback_hours, relevance_filter, enrich_with}`.
- [ ] `manual_events`: optional inline overrides/additions even in non-manual modes (for "real news + hypothetical overlay").
- [ ] **Price resolution** in `oasis/crypto/price_fetch.py`:
  - Priority: `initial_prices[asset]` > `as_of_date` > `price_source=live` > `assets.yaml` default.
  - `fetch_live_prices(symbols)` via `yfinance.download(tickers, period="1d", interval="1m")`, take the most recent close.
  - `fetch_historical_prices(symbols, date)` via `yfinance.Ticker(t).history(start=date, end=date+1d)`.
  - Stablecoin snap: any asset with `class=stablecoin` defaults to `peg_target` (not the fetched value) unless explicitly in `initial_prices`.
  - Cache fetched values to `data/market/price_cache/{YYYY-MM-DD}.json` for air-gapped re-runs.
  - Offline fallback: if live fetch fails → default from `assets.yaml` + warning logged, not crash.
- [ ] Add `yfinance`, `requests` (or `httpx` async), `pydantic` to `pyproject.toml` dependencies.
- [ ] `scripts/run_scenario.py` CLI: loads scenario → resolves prices → resolves news (via news_ingest) → constructs `Simulation` → calls `run()` → dumps parquet results.
- [ ] News-to-sim wiring: resolved news events become `news_event` rows which the info-filter surfaces to eligible agents at the right step (based on `audience` + social graph).
- [ ] Results artifact: `results/<scenario>/<timestamp>/{trades.parquet, orders.parquet, posts.parquet, prices.parquet, actions.parquet, stimuli.parquet, news.parquet, initial_prices.json, config.yaml}`.
- **Verify**: 5 scenarios run end-to-end to completion:
  - `quiet_market.yaml` (no news, mostly-silent agents)
  - `fed_hawkish.yaml` (manual macro news)
  - `kol_pump.yaml` (manual influencer shill)
  - `live_today.yaml` (live-snapshot news + live prices, 60 steps)
  - `calibration/luna_depeg.yaml` (historical news from 2022-05-07..11 with real prices)

#### Phase 10b — Frontend (Next.js 15 + TypeScript + Tailwind)

**Design system** (`frontend/lib/tokens.ts`):
- [ ] Color palette: bg `#0a0e14`, panel `#121823`, border `#1e2838`, text `#e4ecf7`, dim `#7f8ea8`, bull `#00ff88`, bear `#ff3355`, warn `#ffaa00`, cyan `#00ddff`, purple `#a855f7`.
- [ ] **Semantic coloring rule (strict)**: color is meaning, never decoration.
  - `bull` (green) ONLY on positive price deltas, bullish indicators, pass states.
  - `bear` (red) ONLY on negative deltas, bearish indicators, fail states.
  - Neutral/zero data renders in `text` or `dim` — never tinted.
  - `cyan` and `purple` are CHROME ONLY — borders, live-indicator dots, badges, accent lines. Never applied to data values.
  - `warn` (yellow) ONLY for stale data, warnings, borderline states.

### Phase 11 — Eval mode (metrics, baselines, calibration harness)

The eval framework scores simulation output against ground truth (historical) and baselines (always). It is the credibility backbone of the whole tool.

**Eval runs in three sub-modes:**
1. **Historical**: sim vs real price path + real news timing. Used for calibration and hold-out validation.
2. **Sanity**: sim vs baselines (random walk, no-news, shuffled-news, uniform-persona). Used for forward sims too.
3. **Stress**: single-archetype and zero-news runs. Qualitative shape checks (HODLer → flat, FOMO → divergent, Contrarian → oscillatory).

**Metrics (6 tiers — see §7 below for full catalog):**
- [ ] Tier A — Price path: direction-match %, peak-drawdown error, drawdown timing error, recovery half-life error, terminal price error, path correlation, DTW distance.
- [ ] Tier B — Distributional (style facts): return kurtosis, vol clustering ACF(|r|) at lag 1/5/30, realized vol match, green/red day ratio, volume-return correlation.
- [ ] Tier C — Microstructure: active-agent rate per step, trade size distribution, order-book depth evolution, spread distribution.
- [ ] Tier D — Cross-asset: 5×5 correlation matrix Frobenius distance to real, risk-on/off sign agreement, depeg-cascade sign check.
- [ ] Tier E — Social: post volume around news, sentiment-price lead/lag correlation, KOL→follower influence cascade coverage, persona-consistency rate.
- [ ] Tier F — Agent-level: per-archetype P&L distribution, Gini of wealth evolution, action-tier distribution per archetype vs configured base rates.
- [ ] All metrics are computed both directionally (sign match) and absolutely (magnitude error) where applicable.

**Baselines:**
- [ ] `random_walk` — Brownian motion with observed volatility (the floor).
- [ ] `constant` — prices don't move (absolute floor).
- [ ] `replay` — real prices as-is (the ceiling, oracle).
- [ ] `no_agent` — harness runs with one MM, no social agents.
- [ ] `uniform_persona` — all agents are a single archetype.
- [ ] `no_news` — same scenario minus news events.
- [ ] `shuffled_news` — news events at randomized steps.

**Aggregation:**
- [ ] Multi-seed runs: default N=10 per scenario. Report mean ± 95% CI (bootstrap, no normality assumption).
- [ ] `aggregate.py` handles NaN-safe mean, std, quantiles; flags runs with degenerate outcomes (zero trades, degenerate prices).

**Reporting:**
- [ ] `report.py` generates markdown + HTML. Per-tier tables, pass/fail vs thresholds, score vector, baseline comparison, caveats.
- [ ] Plot library: price paths with CI bands, correlation matrices, action histograms per archetype, sentiment-vs-price scatter.

**Calibration harness:**
- [ ] Known-event configs: `calibration/{luna_depeg_may_2022, ftx_collapse_nov_2022, etf_approval_jan_2024, elon_doge_may_2021, covid_crash_mar_2020, threearrow_jun_2022}.yaml`.
- [ ] Hold-out protocol: 3 events for calibration, 3 for hold-out. Freeze config after calibration.
- [ ] `scripts/calibrate.py` — tuning loop that varies archetype ratios / stimulus weights / gate rates, scores each config on calibration set, reports best.

**Thresholds (MVP — initial; refine with real data):**
- [ ] Direction-match % (conditional on |Δreal| > 10 bps): >55% on calibration, >52% on hold-out.
- [ ] Peak drawdown relative error: <0.30 on calibration, <0.50 on hold-out.
- [ ] Path correlation: >0.4 on calibration, >0.2 on hold-out.
- [ ] Must beat `random_walk` on direction-match + drawdown error, p<0.05.
- [ ] Must beat `no_news` on event-window drawdown magnitude, p<0.05.
- [ ] Stress-test qualitative: 100%-HODLer vol < 30% of baseline; 100%-FOMO produces divergence within 2h sim time.

**Ground truth sources (tier-by-tier availability honesty):**
- [ ] **Tier A prices**: Binance public klines API for BTC/ETH/USDT 1-min (free, 2017+, ~10 calls/week per asset). yfinance `GC=F`/`CL=F` for commodities at DAILY resolution — 1-min commodity data is paid (Polygon/Norgate) so MVP reports commodity Tier A at daily with a caveat.
- [ ] **Tier B**: derived entirely from Tier A prices. No extra source.
- [ ] **Tier C**: Binance `/aggTrades` for trade-size distribution and aggregate volume (free). SKIP order book depth (paid: Tardis/Kaiko) in MVP; report only the 3 free metrics.
- [ ] **Tier D**: derived from multi-asset Tier A prices at the coarser resolution (daily, because of commodities).
- [ ] **Tier E** (coarse but honest): Fear & Greed Index from `alternative.me/fng/` (daily, free, 2018+), CryptoPanic vote counts (free), Google Trends via `pytrends` (weekly) as attention proxy. Aggregate sim sentiment to GT resolution for comparison — do NOT invent 1-min sentiment GT.
- [ ] **Tier F**: no external GT needed (internal spec-compliance check). Optional sanity: BTC/ETH on-chain address distributions from Etherscan / BTC-RPC for Gini anchor.
- [ ] Alignment rule: on commodity-market-closed days (weekends/holidays), sim freezes commodity prices to match GT. Documented behavior.
- [ ] All GT cached to `data/market/ground_truth/cache/*.parquet` for air-gapped offline reproduction.

**Forward-sim eval (no ground truth available):**
- [ ] `event_library/` of classified past events with metric signatures (major_bearish_news, stablecoin_depeg, macro_hawkish, etc.).
- [ ] Forward-sim report shows metrics against **percentile bands of similar past events** ("your sim is at the 40th percentile of past major-bearish events").
- [ ] Plausibility pass requires sim metrics within [5th, 95th] percentile of reference distribution.

**Verify:** full eval report on `luna_depeg` replay with real Binance prices. Calibration score vector computed. Baseline comparisons computed. Hold-out score vector computed on FTX collapse. Forward-sim eval runs on `live_today` scenario using event_library plausibility bands. All plots render. Report is readable by a trader who wasn't in the design meetings.

## 7. Eval metric catalog (reference for Phase 11)

Each metric has a type (**D**irectional / **A**bsolute / **X** distributional), a target / threshold, and notes on when to use it.

### Tier A — Price path
| Metric | Type | Definition | MVP threshold |
|---|---|---|---|
| Direction-match % | D | `mean(sign(Δsim) == sign(Δreal))` conditional on `|Δreal| > 10bps` | >55% calib / >52% holdout |
| Peak drawdown error | A | `|max_dd(sim) − max_dd(real)| / max_dd(real)` | <0.30 calib / <0.50 holdout |
| Drawdown timing error | A | `|step_of_min(sim) − step_of_min(real)|` as fraction of duration | <0.10 |
| Recovery half-life error | A | Steps to recover 50% of drawdown | <2× real |
| Terminal price error | A | `|sim_last − real_last| / real_last` | <0.15 |
| Path correlation | D-ish | Pearson corr of price series | >0.4 calib / >0.2 holdout |
| DTW distance | A | Dynamic time-warp distance (timing-tolerant) | Lower = better, no absolute threshold |

### Tier B — Distributional (style facts)
| Metric | Type | Target | Purpose |
|---|---|---|---|
| Return kurtosis | X | Sim in [4, 20] | Fat tails present |
| ACF(\|r\|) @ lag 1 | X | >0.15 | Vol clustering |
| ACF(\|r\|) @ lag 30 | X | >0.05 | Long-memory vol |
| Realized σ match | A | Within 2× of real | Magnitude calibration |
| Green/red ratio | A | Within ±5% of real (crypto ~0.52) | Drift realism |
| Volume-return corr | D | >0.3 | Active trading structure |

### Tier C — Microstructure
| Metric | Type | Target |
|---|---|---|
| Active-agent rate per step | A+D | Tracks real activity index if available; otherwise spikes around news |
| Active-rate-during-news / baseline | A | >2× baseline |
| Trade size distribution | X | Log-normal shape; whale tail at >99th percentile |
| Book depth at spread | A | Collapses >50% during stress, recovers 80% within 30 steps |
| Bid-ask spread distribution | X | Wider mean + fatter tail in stress windows |

### Tier D — Cross-asset
| Metric | Type | Target |
|---|---|---|
| 5×5 correlation matrix Frobenius distance | A | <1.0 (scale-dependent) |
| Risk-on/off sign agreement | D | >80% of macro-shock windows |
| USDT depeg → BTC/ETH sign | D | BTC & ETH drop ≥50bps within 10 steps of USDT depeg >1% |
| Gold-crypto inverse corr on macro shocks | D | Correlation sign flips negative during macro stress |

### Tier E — Social
| Metric | Type | Target |
|---|---|---|
| Post volume around news (±30 steps) | A+D | Spike of ≥3× baseline |
| Sentiment-price lag | D | Sentiment leads or coincides with price; lag ∈ [−5, +5] steps |
| KOL→follower cascade coverage | D | ≥40% of follower-agents act within 5 steps of a KOL post |
| Persona-consistency rate | D | ≥90% of "HODLer" agents hold through ≥20% drawdowns |

### Tier F — Agent-level
| Metric | Type | Target |
|---|---|---|
| Per-archetype P&L distribution shape | X | HODLer: wide positive mean in bull; MM: tight, small +; FOMO: bimodal |
| Gini of final wealth | A | In [0.55, 0.85] for crypto-like concentration |
| Action-tier distribution per archetype | X | Within 2σ of configured base rates over full sim |
| Conservation check | A | Total USD + inventory-at-last-price invariant within 10−6 tolerance |

### Baselines (always computed alongside sim)
| Baseline | What it is | Beat criterion |
|---|---|---|
| `random_walk` | Brownian with observed vol | Sim beats on direction-match + drawdown, p<0.05 |
| `constant` | Flat price | Sim beats on basically everything (sanity) |
| `replay` | Real prices as sim output | Ceiling — sim is at most as good as this |
| `no_agent` | Only a single MM | Sim beats on event-window magnitude |
| `uniform_persona` | All one archetype | Sim beats on style-fact realism |
| `no_news` | Same scenario minus news | Sim beats on event-window drawdown |
| `shuffled_news` | News at random steps | Sim beats on drawdown-timing error |

### Composite score vector

Rather than a single score, report a vector:
```
score_vector = {
  "price_path":      0-1 composite of Tier A,
  "style_facts":     0-1 composite of Tier B,
  "microstructure":  0-1 composite of Tier C,
  "cross_asset":     0-1 composite of Tier D,
  "social":          0-1 composite of Tier E,
  "agent_level":     0-1 composite of Tier F,
  "vs_random_walk":  composite advantage over random walk baseline,
  "vs_no_news":      composite advantage over no-news baseline,
}
```

Different research questions weight these differently:
- "Is our simulation useful for trader scenarios?" → weight Tier A + Tier D + vs_random_walk.
- "Is the narrative engine doing anything?" → weight Tier E + vs_no_news.
- "Are our personas coherent?" → weight Tier F.
